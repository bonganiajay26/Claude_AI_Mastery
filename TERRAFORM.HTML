📄 azure-pipelines.yml (Key Stages)
trigger:
  branches: { include: [main, staging] }
  paths: { include: [terraform/**] }

stages:
  - stage: Validate
    jobs:
      - job: TerraformValidate
        steps:
          - task: TerraformInstaller@0
          - script: terraform init -backend=false
          - script: terraform validate && terraform fmt --check
          - script: checkov -d . --framework terraform

  - stage: Plan
    jobs:
      - job: TerraformPlan
        steps:
          - task: TerraformTaskV4@4
            inputs: { command: init, backendType: azurerm }
          - task: TerraformTaskV4@4
            inputs: { command: plan, commandOptions: '-out=tfplan' }

  - stage: Apply
    condition: and(succeeded(), eq(variables['Build.SourceBranch'], 'refs/heads/main'))
    jobs:
      - deployment: TerraformApply
        environment: production   # manual approval gate
        strategy:
          runOnce:
            deploy:
              steps:
                - task: TerraformTaskV4@4
                  inputs: { command: apply, commandOptions: 'tfplan' }

📄 buildspec.yml (AWS CodeBuild)
version: 0.2

env:
  parameter-store:
    TF_VAR_db_password: /prod/db/password
    AWS_ACCESS_KEY_ID:  /terraform/access-key

phases:
  install:
    commands:
      - wget https://releases.hashicorp.com/terraform/1.8.0/terraform_1.8.0_linux_amd64.zip
      - unzip terraform*.zip && mv terraform /usr/local/bin/
      - pip install checkov

  pre_build:
    commands:
      - terraform init -backend-config="bucket=$TF_STATE_BUCKET"
      - terraform validate && terraform fmt -check
      - checkov -d . --framework terraform --output junitxml
      - tfsec . --format json

  build:
    commands:
      - terraform plan -out=tfplan -var-file=environments/$ENV/terraform.tfvars
      - terraform show -json tfplan > tfplan.json
      - aws s3 cp tfplan s3://$TF_STATE_BUCKET/plans/$CODEBUILD_BUILD_ID/

  post_build:
    commands:
      - terraform apply -auto-approve tfplan   # only in Apply stage

artifacts:
  files: [tfplan, tfplan.json, '**/*.xml']

Terraform State Backend — Azure vs AWS

STATE MANAGEMENT

☁️ Azure Blob Storage Backend

🗃️Resource Group → Storage Account → Container
🔒Azure AD / Service Principal auth
🔐Encryption with Azure Key Vault CMK
🔁State locking via Blob lease mechanism
📜Soft-delete + versioning for state history
🌍GRS replication for DR
terraform {
  backend "azurerm" {
    resource_group_name  = "tf-state-rg"
    storage_account_name = "tfstatestorage"
    container_name       = "tfstate"
    key                  = "prod/terraform.tfstate"
    use_oidc             = true
  }
}

🟠 AWS S3 + DynamoDB Backend

🗃️S3 bucket with versioning enabled
🔒IAM Role / OIDC for pipeline auth
🔐S3 SSE-KMS encryption
🔁DynamoDB table for state locking
📜S3 Versioning for state history
🌍Cross-region replication for DR
terraform {
  backend "s3" {
    bucket         = "my-tf-state-bucket"
    key            = "prod/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-state-lock"
    encrypt        = true
    kms_key_id     = "alias/terraform-state"
  }
}

End-to-End Workflow (10 Steps)

FULL FLOW
01

👨‍💻 Developer Writes Terraform Code

Engineer creates/modifies .tf files in a feature branch. Uses VS Code with HashiCorp Terraform extension. Follows module structure conventions.

git checkout -b feature/add-rds
02

🔍 Pre-commit Hooks Run Locally

terraform fmt, terraform validate, tflint, and detect-secrets hooks run before commit. Catches issues before CI even starts.

pre-commit run --all-files
03

📤 PR Opened → CI Triggers

Pull Request opened against main. Azure Pipelines / CodePipeline trigger automatically. Pipeline validates code, runs security scans with checkov and tfsec.

checkov -d . --framework terraform
04

🏗️ terraform init + Backend Config

Pipeline initialises Terraform with remote backend. Downloads providers, configures Azure Blob or S3+DynamoDB state. Service Principal / IAM Role auth.

terraform init -reconfigure
05

📋 terraform plan — Change Preview

Generates execution plan. Shows exactly what will be created/modified/destroyed. Plan saved as artifact and published as PR comment for review.

terraform plan -out=tfplan.binary
06

👁️ Manual Approval Gate (Prod)

For staging/prod: senior engineer reviews plan output. Azure DevOps Environments approval, or AWS SNS + Lambda approval gate. No auto-apply in production.

Approval required — Azure Env Gate
07

🚀 terraform apply — Resources Created

Approved plan executed. Terraform provisions/modifies cloud resources. State written to remote backend. Outputs captured and passed to downstream pipelines.

terraform apply tfplan.binary
08

🔒 Secrets Injected via Vault / KMS

No secrets in code. Azure Key Vault or AWS Secrets Manager referenced via TF data sources. OIDC token-based auth — no long-lived keys in pipelines.

data "azurerm_key_vault_secret" {}
09

🔄 Drift Detection (Scheduled)

Scheduled pipeline runs terraform plan daily. Any drift (manual changes in console) alerts the team via Slack/Teams/PagerDuty. Drift auto-remediation optional.

cron: "0 6 * * *" # daily at 6am
10

📊 Observability & Cost Governance

Azure Monitor / CloudWatch track provisioned resources. Infracost estimates cost delta per PR. Resource tags enforced via TF policy. Budget alerts configured.

infracost diff --path=.

Multi-cloud Terraform IaC — optimised for DevOps engineers
🏗️ This is how production teams manage Terraform at scale across Azure AND AWS.

Most engineers treat infrastructure like a one-time setup. The best teams treat it like software — versioned, reviewed, tested, and automated.

Here's the full architecture 👇

✅ Terraform as the single IaC engine for both Azure and AWS
✅ Azure Pipelines / CodePipeline for automated plan → approve → apply
✅ Remote state in Azure Blob / S3 with DynamoDB locking — zero conflicts

✅ Azure Key Vault / Secrets Manager — zero hardcoded credentials. Ever.

The golden rule: If it's not in Git, it doesn't exist in production.
